# Prompt Design Notes

这份文档用于总结当前项目中 `Ask` / `Action` / `Summary` / `loop` / `tools` 的设计过程、关键决策和踩坑经验，方便迁移到其他项目后继续迭代提示词。

当前项目在业务层会拆成 3 种模式，每种模式都各自拥有 `Ask` 和 `Action` 两阶段：

- `Connector`：用于广泛寻找并倾向互动的模式，也是当前这套提示词流程对应的模式
- `Engager`：用于围绕一个明确用户名，收集目标用户直接信息，以及 `X` / `SparkRizz` 内与其相关内容，并可按用户偏好决定是否互动推进的模式
- `Finder`：用于寻找目标人群或候选人物，且明确禁止发帖、提及、回复等互动行为、只专注找人的模式

当前目录下已经落地的文档名称分别是：

- 通用 Ask 提示词：`common-ask.md`
- Connector Ask 提示词：`connector-ask.md`
- Engager Ask 提示词：`engager-ask.md`
- Finder Ask 提示词：`finder-ask.md`
- 通用 Action 提示词：`common-action.md`
- Connector Action 提示词：`connector-action.md`
- Engager Action 提示词：`engager-action.md`
- Finder Action 提示词：`finder-action.md`
- 社交任务 Summary 提示词：`common-summary.md`
- Planner / Executor / Replanner 的调度规则已内置在 `agenttask.go`

系统在加载这些提示词进入上下文时，Ask / Action 之间仍以固定代号互相引用：

- Ask 提示词在上下文中使用 `ASK.md` 作为代号
- Action 提示词在上下文中使用 `ACTION.md` 作为代号

也就是说，文件真实命名按业务模式区分，但提示词内部互相引用时，仍应使用 `ASK.md` / `ACTION.md`。

对于澄清阶段，运行时会把 `common-ask`、对应模式的 `ask` prompt 直接拼接后一起注入上下文。

- 因此模式级 `ask` prompt 不需要特意点名 `common-ask.md`
- 模式级 `ask` prompt 只需要直接补充自己的差异化澄清重点
- 通用的提问方法、工具规则和 `$WANT` 输出格式由拼接进来的上游 prompt 自然提供

对于执行阶段，运行时会注入 `Planner` / `Executor` / `Replanner` 的专用 system prompt，同时把 `common-action`、对应模式的 `action` prompt 直接拼接后一起注入上下文。

- 因此模式级 `action` prompt 不需要特意点名 `common-action.md`，也不需要重复解释调度层的 Planner / Replanner 规则
- 模式级 `action` prompt 只需要直接补充自己的差异化规则
- 通用执行纪律和调度层继续 / 结束规则由上游 prompt 自然提供

对于新一轮执行开始前，运行时会重新注入 system prompt，并把上一轮除 system prompt 之外的上下文先压缩为一份 Summary，再作为新轮次最开始的上下文继续使用。

- 因此 `Summary` 的职责不是重复解释 `ACTION.md`、Planner / Replanner 等稳定规则
- `Summary` 只需要保留跨轮继续执行必须依赖的事实状态、对象状态、互动状态和阶段汇报状态
- `Summary` 应服务“下一轮直接接力”，而不是服务人类阅读时的泛化概述

## 1. 系统目标

这套 Agent 不是一次性对话助手，而是一个可跨多轮持续执行的社交任务 Agent。

每个业务模式都分为两个阶段：

- `Ask`：澄清用户意图，产出多行 `$WANT`
- `Action`：基于 `$WANT` 和工具持续执行社交动作，按轮次推进，直到满足结束条件

设计重点不是让模型“会说”，而是让模型：

- 能稳定收集执行所需信息
- 能在多轮任务里保持正确的继续/结束判断
- 能用工具表达清晰的阶段性进展

本文当前详细记录的 Ask / Action 设计经验，主要来自已经落地的 `Connector` 模式；后续补充 `Engager` 和 `Finder` 时，应尽量复用同一套阶段拆分与文档职责，但要保留各模式自己的硬边界，尤其 `Engager` 必须先拿到明确用户名并确认互动偏好、`Finder` 不能回退成可互动模式。

## 2. Ask 阶段设计

### 2.1 Ask 的职责

Ask 只负责澄清意图，不负责执行动作。

明确排除了这些行为：

- 不代替用户发帖
- 不代替用户搜索
- 不代替用户回复
- 不执行任何具体社交动作

这样做的原因是避免 Ask 和 Action 职责混淆，否则模型容易在澄清阶段提前开始“做事”。

### 2.2 Ask 文档拆分

Ask 相关规则现在也拆成两层：

- `common-ask.md`：负责所有模式共享的澄清方法与 `$WANT` 输出规则
- `<mode>-ask.md`：负责各模式自己的澄清重点

这里的核心原则是：

- 通用的提问方法、工具使用规则、`$WANT` 输出格式，应尽量收敛到 `common-ask.md`
- 各模式 Ask 文档只补充该模式下真正需要额外澄清的信息和输出偏向
- 不要在每个模式 Ask 文档里重复同一套通用追问规则，否则后续非常容易漂移
- 模式级 Ask 文档不需要特意引用 `common-ask.md` 的文件名，因为运行时会直接拼接这些内容

### 2.3 Ask 的输出不是 key-value，而是多行 WANT

最终确定 `$WANT` 不再要求 key-value 字段，而是输出带明确倾向的多行结果。这些行合起来至少要覆盖：

- 想接触什么人
- 想达成什么结果
- 更偏向搜寻，还是更偏向主动发布内容吸引人
- 是否允许主动发帖
- 有哪些边界、禁忌、不能做的事
- 什么结果算成功
- 达到什么程度可以结束任务

这样设计的原因：

- `Action` 需要的是可执行决策信息，而不是模糊总结
- 多行倾向结果比 key-value 更贴近模型自然表达，稳定性更高
- 只保留执行真正需要的意图判断，迁移到其他项目时更容易改造

### 2.4 Ask 工具设计

Ask 阶段围绕 `generate_want` 工作。

规则是：

- 信息足够时，调用 `generate_want.output` 产出多行 `$WANT`
- 信息不足时，调用 `generate_want.ask` 继续追问

这里踩过一个坑：

- 一开始工具描述残留了 `plan` / “步骤列表” 的语义，导致工具心智不统一
- 之后统一改成“生成 WANT 或继续追问”

另一个坑是：

- 最初把 `output` 误写成必填字段
- 这会与“信息不足时只继续追问”的逻辑冲突

因此迁移时要注意：

- Ask 工具要支持“生成结果”和“继续追问”两种分支
- 不要把 Ask 工具误做成固定计划生成器
- 生成结果时，`output` 应是多行倾向文本，而不是 key-value 列表

### 2.5 Ask 提问策略

最终保留的策略是：

- 默认一次只问一个问题
- 如果多个信息点强相关，可一次问最多 3 个关键问题
- 总问题数尽量控制在 5 个以内

这里的目标不是压缩到绝对最少轮，而是避免用户一次性输入过多内容，同时保证得到可执行信息。

## 3. Action 阶段设计

### 3.1 Action 不再反复确认用户

一旦 `$WANT` 足够清晰，Action 应自主执行，不再频繁向用户确认。

这是因为这类任务是长任务，如果每一步都确认，Agent 会丧失自动推进能力。

### 3.2 Action 文档拆分

Action 相关规则现在拆成两层：

- `common-action.md`：负责所有模式共享的执行纪律
- `<mode>-action.md`：负责各模式自己的行为差异

这里的核心原则是：

- 通用的轮次规则、汇报规则、结束任务规则，应尽量收敛到 `common-action.md`
- 各模式的 Action 文档只补充该模式下“做什么、优先做什么、什么结果更有价值”
- 不要在每个模式 Action 文档里重复定义同一套通用结束纪律，否则后续非常容易漂移
- 模式级 Action 文档不需要特意引用 `common-action.md` 或调度层 prompt 的文件名，因为运行时会直接注入这些内容

### 3.3 行为能力边界

最终只明确保留了当前工具支持的能力：

- 主动发帖（包含提及）
- 搜寻
- 回复提及过我的帖子

并明确要求：

- 工具不支持的动作不能自行假设能执行

这里的原因是避免模型脑补出 DM、关注、连续骚扰等未提供能力。

### 3.4 行为优先级

没有把流程写死，但给了明确倾向：

1. 先搜索
2. 再基于上下文主动发帖
3. 最后才提及具体用户

这样做是为了避免模型一上来就主动打扰别人，同时保留策略灵活性。

### 3.5 价值判断标准

最终保留三项核心判断维度：

- 与 `$WANT` 的相关性
- 回复或互动质量
- 对方真实性

原因是很多结束条件和汇报条件都依赖“有价值”，如果不定义，模型会把低质量互动误判成成果。

## 4. 多轮任务设计

### 4.1 这是一个默认跨多轮推进的任务

这是整个设计里最关键的一条。

最早的版本虽然写了 `loop`，但模型仍然会出现：

- 做完一轮动作后直接结束任务
- 发完帖后立刻 `finish_task`

所以后来专门强化成硬规则：

- 这是默认跨多轮推进的任务
- 完成某一个动作不等于任务完成
- 只要仍存在待观察的外部反馈窗口，就应继续后续轮次

### 4.2 结束任务的硬门槛

后来又把结束任务进一步收紧成更硬的规则：

- `complete_agent_task` 是真正完成任务后的退出流程，不能轻易调用
- `loop < 5` 时绝对禁止结束任务
- 即使第 2 到第 4 轮已经拿到强结果，也不能提前结束

这样做的原因是：

- 只要保留“强结果可提前结束”的例外，模型就很容易把阶段性成果误判成最终完成
- 用轮次硬门槛约束，比写一堆软性解释更稳定
- 这样可以把“是否允许评估结束”和“为什么可以结束”清晰拆开

### 4.3 前 4 轮的推进策略

在前 4 轮不能结束的前提下，还需要补一条对冲约束：

- 前 4 轮不能为了凑轮次而重复发帖、重复提及、重复骚扰
- 如果进展较弱，应优先做轻量观察、补充搜索、整理候选人、等待反馈

这条规则是为了避免模型把“不能结束”错误理解成“必须继续刷动作”。

### 4.4 首次启动汇报

进入 `Action` 阶段后，还额外加了一条启动期规则：

- 如果尚未发送过首次启动汇报，则应在第 1 轮尽早发送一次 `send_progress_report_message`
- 首次启动汇报至少要说明：已进入执行阶段、当前如何理解 `$WANT`、本轮准备先做什么
- 如果第 1 轮已经拿到值得汇报的结果，可以把启动汇报和结果汇报合并成一条

这样做的原因是避免用户在任务刚进入执行时长时间看不到任何反馈。

### 4.5 loop 规则的职责

这部分规则现在由 `Planner` / `Replanner` 的专用 system prompt 承担，负责“继续还是结束”的主判断。

它明确区分了两类情况：

应该继续后续轮次：

- 当前还未进入第 5 轮
- 刚做完需要等待反馈的动作
- 发现了还值得继续观察的线索
- 仍存在待观察的反馈窗口

更适合结束任务：

- 当前已进入第 5 轮或更后续轮次
- 已拿到高价值回复或明确结果
- 多次系统唤醒后，当前积累结果比继续等待更有价值
- 达到时间上限
- 连续多个轮次没有新增高价值线索

迁移时建议继续沿用这个分工：

- `common-action.md` 负责通用执行纪律
- `connector-action.md` 负责 Connector 模式下“如何做”
- `Planner` / `Replanner` 负责“何时继续 / 何时结束”

同时要把结束判断拆成两层：

- `结束资格`：是否已经具备评估结束的资格，由 `Planner` / `Replanner` 约束
- `结束理由`：在已经具备结束资格后，因为什么结果可以结束，由 `connector-action.md` 约束

这两层不能混在一起。没有结束资格时，即使已经拿到阶段性成果，也不能调用结束工具。

## 5. Summary 阶段设计

### 5.1 Summary 的职责不是“泛化摘要”

`Summary` 最初很容易被写成通用任务摘要器，但在当前这类跨多轮社交任务里，这样不够。

真正需要的是：

- 一份可跨轮恢复的状态快照
- 一份可追溯的对象与互动记录
- 一份让下一轮 Agent 无需回看长历史也能继续执行的工作记忆

也就是说，`Summary` 的核心职责不是“写短”，而是：

- 保留下一轮必需状态
- 删除低价值噪音
- 避免因上下文压缩而丢失后续判断所依赖的关键事实

### 5.2 Summary 在运行链路中的位置

当前运行链路里，`Summary` 的位置应理解为：

1. 系统开始新轮次时，重新注入 `ACTION.md`、Planner / Executor / Replanner 等 system prompt  
2. 上一轮除 system prompt 之外的上下文，不直接整段带入  
3. 这些上下文先经过 `Summary` 压缩  
4. `Summary` 输出结果作为新轮次最开始的一段上下文提供给下一轮 Agent

因此这条链路决定了：

- `Summary` 不需要重复 system prompt 中已稳定存在的规则
- `Summary` 必须保留上一轮真实新增的业务状态
- `Summary` 必须足够完整，避免下一轮重新搜索、重复发帖、重复汇报

### 5.3 Summary 要保留什么

目前已经明确，以下信息是跨轮必须重点保留的：

- 高潜质用户的完整档案
- 已看过的重要帖子
- 当前用户自己已经发过的帖子
- 已发生的关键互动及待观察窗口
- 已发送给用户的阶段性汇报

这里踩出的经验是：

- 如果只保留“发现了几个高潜用户”这种概括，下一轮无法继续判断谁值得跟进
- 如果不保留帖子完整内容，下一轮很难判断是否已经看过、为什么重要、是否值得再次互动
- 如果不保留自己发过的帖子及其反馈状态，下一轮容易重复发帖或漏看回复
- 如果不保留阶段性汇报中的 `next_step`，下一轮容易偏离已经向用户承诺的后续动作

因此，Summary 对关键对象默认应尽量保留：

- 完整用户标识：`id`、`uid`、用户名、展示名等
- 完整帖子标识：`post_id`、作者信息、标题、正文内容
- 完整互动关系：谁对谁做了什么、哪些结果仍在等待观察
- 完整汇报内容：`title`、`content`、`next_step`

### 5.4 Summary 不应该暴露业务模式名

虽然当前这套流程落在 `Connector` 模式下，但 `Summary` 提示词本身不需要把 `Connector` 作为任务名显式暴露给 Agent。

这里的考虑是：

- `Summary` 的职责是总结“社交任务”的运行态状态
- 模式差异更多体现在 Ask / Action 的具体行为规则里
- 在 Summary 层强调模式名，对状态恢复本身没有帮助，反而会增加无关表述

因此当前更合适的写法是：

- 文件名统一为 `common-summary.md`
- 但提示词正文内使用“社交任务”描述当前任务，而不是反复强调 `Connector`

## 6. 结束工具设计

### 6.1 为什么弃用“状态更新”心智

中途尝试过把结束工具设计成状态更新类工具：

- `update_agent_task_status`

问题是：

- 一旦暴露 `FAILED` 枚举，模型就会把很多正常业务收口误判成失败
- 即使工具说明里强调“别用 failed”，模型仍然可能受枚举影响

因此最终设计回到更简单的心智：

- 给模型一个唯一明确的正常出口

最终选择：

- `complete_agent_task`

参数只保留：

- `agent_task_id`

设计原则是：

- 模型不感知失败枚举
- 模型只知道“当任务满足结束条件时，调用这个工具”

这里后来又补出一个关键经验：

- 模型容易把“已经做过动作”误判成“任务已经完成”
- 模型也容易把“已经有可交付结果”误判成“现在就可以收尾”

因此只强调“命中结束条件再结束”还不够，还必须单独约束：

- 首先判断是否具备结束资格
- `loop < 5` 时绝对不允许认真评估结束
- 前 4 轮拿到的候选人、线索、初步互动或阶段性成果，默认都不是最终完成

这是减少逻辑打架最有效的方式。

## 7. 阶段性汇报工具设计

### 7.1 为什么不用泛化的 send_message

最初通知工具是：

- `notify_user`
- 后来改成 `send_message`

但这两个名字都太宽泛，容易让模型把它当普通聊天出口。

最终改为：

- `send_progress_report_message`

这样可以更明确约束它的用途：

- 不是随便发消息
- 而是发送“阶段性汇报”

### 7.2 什么时候应该汇报

汇报触发条件最终明确为：

- 刚进入执行阶段，需要先给用户一次启动反馈
- 发现有价值的人
- 发现有价值的内容
- 出现新的互动结果
- 任务进入了值得通知用户的新阶段

### 7.3 汇报风格要求

最终确定的风格不是长段说明，而是：

- 多行短句
- 突出本轮发现
- 说明为什么有价值
- 告诉用户下一步准备做什么

并在提示词里保留了三种汇报重点：

- 发现了有价值的人
- 发现了有价值的内容
- 阶段性进展

这里有一个重要决策：

- 对应模式的 Action 提示词在上下文中仍以 `ACTION.md` 代号出现，并且只保留“什么时候汇报、重点偏什么”
- `title / content / next_step` 等字段语义交给工具定义去说明

这样可以减少重复，避免提示词层和工具层重复解释同一件事。

## 8. 文档拆分原则

当前文档职责建议保持如下：

- `common-ask.md`：负责所有模式共享的澄清方法与 `$WANT` 输出规则
- `<mode>-ask prompt`：真实文件名按模式区分，但进入上下文后仍以 `ASK.md` 代号使用，负责补充该模式下需要澄清的差异信息
- `common-action.md`：负责所有模式共享的执行纪律，包括轮次、首次启动汇报、结束任务约束
- `<mode>-action prompt`：真实文件名按模式区分，但进入上下文后仍以 `ACTION.md` 代号使用，负责该模式的执行目标、行为边界、价值判断、结果优先级
- `common-summary.md`：通用 Summary 提示词，负责把上一轮除 system prompt 外的上下文压缩为可跨轮恢复的状态快照，重点保留实体、帖子、互动和阶段汇报
- `agenttask.go` 中的 Planner / Replanner prompt：跨轮运行逻辑、继续与结束判断
- `tools.go`：工具的输入输出语义、字段约束、使用场景

迁移时不要轻易把这些职责揉在一起，否则模型更容易抓不住重点。

## 9. 已确认的关键决策

以下是本项目里已经明确拍板的决策，迁移时可直接作为默认值：

- APP 业务层按 `Connector` / `Engager` / `Finder` 3 种模式拆分
- 每种模式都拆成各自独立的 `Ask` 与 `Action` 两阶段
- 当前已落地的提示词流程属于 `Connector` 模式
- 通用澄清规则抽到 `common-ask.md`
- 通用执行纪律抽到 `common-action.md`
- 新轮次开始前，需要用 `Summary` 压缩上一轮非 system prompt 上下文，再作为新轮次起始上下文
- Ask 产出多行 `$WANT`，而不是 key-value 结构
- 进入 `Action` 后，第 1 轮应尽早发送首次启动汇报
- `loop < 5` 时绝对禁止结束任务
- 执行优先级倾向为：搜索 -> 发帖 -> 提及
- 有价值的判断维度是：相关性、互动质量、真实性
- 只要存在待观察反馈窗口，就应继续多轮执行
- Agent 的正常结束工具是 `complete_agent_task`
- Agent 的阶段性汇报工具是 `send_progress_report_message`
- `Summary` 不是通用短摘要，而是跨轮状态快照
- `Summary` 不需要总结当前是第几轮，轮次由系统注入
- `Summary` 需要完整保留高潜用户、重要帖子、自己已发帖子、关键互动和阶段性汇报中的 `next_step`
- `Summary` 提示词正文应使用“社交任务”描述任务，而不是显式暴露 `Connector` 名称
- 提示词里提到“平台”时，默认仅指 `X` 与 `SparkRizz`，不能扩展到其他平台
- 如果用户或上下文里出现“本APP”“端内”“spark_rizz”或 `sparkrizz`，内部都应归一理解为当前 APP，即 `SparkRizz`
- 如果需要对用户外显平台名，一律写作 `SparkRizz`
- `Engager` 的 Ask 阶段必须先拿到一个明确、可识别的具体用户名，否则不能产出进入 Action 阶段的 `$WANT`
- `Engager` 的 Ask 阶段应进一步确认用户是否允许互动，以及互动边界
- `Engager` 的 Action 阶段必须先围绕目标用户本人信息和 `X` / `SparkRizz` 相关内容收集展开；只有在 `$WANT` 明确允许互动时，才可以补充互动推进逻辑
- 由于回复能力受平台约束、只能回复被提及后的帖子，`Engager` 在允许互动时的默认优先级应是先发帖或提及目标用户，再视后续上下文决定是否回复
- `Finder` 的定位是“只找人，不互动”，不是 `Connector` 的简单弱化副本；其提示词必须把搜寻、筛选、整理候选人作为主目标，并明确禁止发帖、提及、回复和其他互动推进动作

## 10. 迁移到其他项目时优先检查的点

如果把这套设计迁移到其他 Agent 项目，优先检查这些点：

1. 新项目的 `$WANT` 是否需要新增倾向维度  
例如平台、语言、地域、语气、人群筛选条件等，是否需要体现在多行结果里。

2. 工具能力边界是否变化  
如果新项目多了私信、关注、评论等能力，对应模式的 `action prompt` 必须同步调整。

3. 任务是否仍然是跨多轮任务  
如果是短任务，Planner / Replanner 中的多轮逻辑要弱化；如果仍然是长任务，应继续保留当前硬约束。

4. 汇报是否仍然是阶段性汇报  
如果新项目的消息工具语义不同，不要直接复用 `send_progress_report_message` 的名称和描述。

5. Summary 是否仍然需要“状态快照”而不是“普通摘要”  
如果新项目同样是跨多轮、跨唤醒持续执行，就应继续优先保留实体状态、对象关系和待观察窗口，而不是只保留简短概述。

6. 结束条件是否仍以“高价值回复”优先  
如果新项目更看重线索积累而不是回复，`P0/P1/P2` 需要重排。

## 11. 当前版本仍值得继续观察的点

虽然当前版本已经比最早稳很多，但后续实测仍建议重点观察：

- 模型是否仍会在“首轮主动发帖后”提前结束任务
- 模型是否会把低质量互动误判为高价值结果
- 模型是否会过度频繁地发送阶段性汇报
- 模型是否会把“发现线索”与“已达到结束条件”混淆
- 模型在 `Summary` 阶段是否仍会遗漏高潜用户 ID、帖子全文、已发帖子状态或阶段汇报 `next_step`

如果再出现这些问题，优先从下面两个方向继续收口：

- 把工具描述写得更硬，而不是一味增加提示词段落
- 把结束规则继续集中在 Planner / Replanner，避免 Action 和调度层双重解释
